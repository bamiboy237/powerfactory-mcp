# Buildout Dependency and Evidence Matrix

This matrix is the operational completion gate for [roadmap Buildouts 0–17](../../PRODUCT_ROADMAP.md#9-sequential-product-buildouts) and the [companion specification gate](../../PRODUCT_ROADMAP.md#companion-specification-gate). It records acceptance obligations; it does not replace buildout or specification prose.

## Status and evidence convention

- Status is one of `NOT RUN`, `BLOCKED`, `READY`, `RUNNING`, `PASS`, `FAIL`, or `N/A — <reason>`. Every row starts `NOT RUN`.
- `[NOW]` means the command exists when this matrix is created. `[REQUIRED]` means the buildout must create the referenced script/test path before completion; a missing command is `BLOCKED`, never `PASS`.
- Set `PASS` only after every listed command ran successfully at the recorded commit. An unexecuted, skipped, or locally unavailable command remains `NOT RUN`/`BLOCKED`; do not infer success from code review.
- In the row's evidence record, append the UTC time, commit SHA, exact command, exit code, environment/fixture ID, and artifact path plus SHA-256. Redact licences, credentials, installation secrets, and confidential model data. Reset a stale `PASS` to `READY` when relevant code, specifications, fixtures, or compatibility claims change.
- A specification key below is satisfied only by an accepted, versioned artifact under `specs/`; draft prose is not acceptance evidence.

Specification keys: `DEP` deployment/process ownership; `AUTH` approval authority; `LEASE` context lease/fencing; `ID` identity/revision; `WF` workflow transition table; `REC` crash-reconciliation matrix; `AREA` `area-load-scaling/v1`; `LF` `load-flow-and-violation-policy/v1`; `DOM` machine-readable domain schemas; `MCP` versioned MCP request/response schemas. Every key is governed by the [companion specification gate](../../PRODUCT_ROADMAP.md#companion-specification-gate).
